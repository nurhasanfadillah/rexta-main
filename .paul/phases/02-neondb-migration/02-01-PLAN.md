<objective>
## Goal
Setup NeonDB schema dan Better Auth sebagai fondasi migrasi dari Supabase — tanpa menyentuh app yang sedang berjalan.

## Purpose
Plan ini adalah layer pertama dari 3 plan migrasi. CRUD API (Plan 02-02) dan frontend update (Plan 02-03) tidak bisa berjalan sebelum schema dan auth terkonfigurasi. Supabase tetap berjalan paralel selama proses ini.

## Output
- `db_schema_neon.sql` — schema siap dijalankan di NeonDB dashboard (app tables + Better Auth tables)
- `lib/db.ts` — shared NeonDB client: Pool untuk Better Auth + neon() HTTP adapter untuk CRUD
- `lib/auth.ts` — Better Auth dikonfigurasi dengan NeonDB adapter, trustedOrigins, dan secret aman
- `api/auth/[...all].ts` — Vercel Function catch-all untuk semua auth endpoints
- `api/health.ts` — Vercel Function untuk verifikasi DB connectivity post-deploy
- `vercel.json` — dipatch agar request ke `/api/*` tidak tertangkap SPA rewrite
- `.env.example` — dokumentasi semua environment variables yang dibutuhkan
- `.gitignore` — dipastikan melindungi file credentials dari ter-commit
</objective>

        "userAgent" text,

        "identifier" text NOT NULL,

        "expiresAt" timestamp NOT NULL,

    3. Buat `lib/db.ts` — dual NeonDB client (Pool + HTTP adapter): <!-- audit-added -->
       ```typescript
       import { Pool, neon } from '@neondatabase/serverless';

       // Pool untuk Better Auth (butuh pg-compatible Pool interface)
       // Connection limits wajib — NeonDB free tier max 100 connections.
       // Tanpa limits, invocation paralel bisa exhausted.
       export const db = new Pool({
         connectionString: process.env.DATABASE_URL,
         max: 10,
         connectionTimeoutMillis: 5000,
         idleTimeoutMillis: 10000,
       });

       // HTTP adapter untuk CRUD queries di Plan 02-02.
       // Stateless, tidak perlu koneksi persisten — optimal untuk serverless.
       export const sql = neon(process.env.DATABASE_URL!);
       ```

    4. Buat `lib/auth.ts` — Better Auth config dengan CSRF protection: <!-- audit-added -->
       ```typescript
       import { betterAuth } from 'better-auth';
       import { Pool } from '@neondatabase/serverless';

       if (!process.env.BETTER_AUTH_URL) {
         throw new Error('BETTER_AUTH_URL env var wajib di-set. Jangan gunakan localhost di production.');
       }

       if (!process.env.BETTER_AUTH_SECRET) {
         throw new Error('BETTER_AUTH_SECRET env var wajib di-set.');
       }

       export const auth = betterAuth({
         database: new Pool({
           connectionString: process.env.DATABASE_URL,
         }),
         emailAndPassword: {
           enabled: true,
         },
         secret: process.env.BETTER_AUTH_SECRET,
         baseURL: process.env.BETTER_AUTH_URL,
         trustedOrigins: [process.env.BETTER_AUTH_URL],
       });
       ```
       PENTING: `baseURL` tidak boleh ada fallback ke localhost. Jika env var tidak di-set,
       sistem harus gagal dengan pesan error yang jelas, bukan diam-diam pakai localhost.

    5. Buat `.env.example` di root project:
       ```
       # NeonDB Connection (Pooler URL — untuk queries)
       DATABASE_URL=postgresql://username:password@host/dbname?sslmode=require&channel_binding=require

       # Better Auth — SECRET WAJIB RANDOM, bukan kata-kata manusia
       # Generate dengan: openssl rand -base64 32
       # Contoh output: K7vP2mXqR9nL4wJhY1cF8sB6tA3dE0gN5iM+uQzW=
       # JANGAN gunakan password atau kata-kata yang mudah ditebak
       BETTER_AUTH_SECRET=hasil-output-openssl-rand-base64-32

       # URL aplikasi tanpa trailing slash
       BETTER_AUTH_URL=https://rexta.redone.my.id

       # Supabase (DEPRECATED — akan dihapus setelah Plan 02-03 selesai)
       VITE_SUPABASE_URL=
       VITE_SUPABASE_ANON_KEY=
       ```

  <name>Task 3: Buat auth Vercel Function + health endpoint + fix vercel.json</name>
  <files>api/auth/[...all].ts, api/health.ts, vercel.json</files>
  <action>
    1. Buat direktori `api/auth/` di root project.

    2. Buat `api/auth/[...all].ts`:
       ```typescript
       import { toNodeHandler } from 'better-auth/node';
       import { auth } from '../../lib/auth';

       export default toNodeHandler(auth);
       ```
       File ini adalah catch-all handler — Vercel akan route semua `/api/auth/*` ke sini.
       `toNodeHandler` mengkonversi Better Auth handler ke format Vercel Function (Node.js).

    3. Buat `api/health.ts` — DB connectivity check untuk verifikasi post-deploy: <!-- audit-added -->
       ```typescript
       import { db } from '../lib/db';

       export default async function handler(req: any, res: any) {
         try {
           await db.query('SELECT 1');
           res.json({ status: 'ok', db: 'connected', timestamp: new Date().toISOString() });
         } catch (error) {
           console.error('[health] DB connection failed:', error);
           res.status(500).json({ status: 'error', db: 'disconnected' });
         }
       }
       ```
       Endpoint ini tidak butuh auth — hanya cek apakah koneksi ke NeonDB berhasil.
       Gunakan ini segera setelah deploy untuk verifikasi sebelum lanjut ke Plan 02-02.

    4. Update `vercel.json` — fix SPA rewrite agar tidak catch API routes:
       ```json
       {
         "rewrites": [
           { "source": "/((?!api/).*)", "destination": "/index.html" }
         ]
       }
       ```
       Regex `(?!api/)` adalah negative lookahead: semua path KECUALI yang dimulai dengan
       `api/` akan di-rewrite ke index.html. Vercel Functions di `api/` tetap berjalan normal.

    Avoid: Jangan buat `api/index.ts` atau handler lain yang bisa conflict.
    Avoid: Jangan hapus rewrite yang ada — hanya ubah `source` pattern-nya.
    Avoid: Jangan import dari `services/` di file `api/` — gunakan `lib/` saja.
    Avoid: Jangan tambahkan auth check di `/api/health` — ini harus public untuk monitoring.
  </action>
